This guide walks you through installing franz-go and writing a program that produces records to a Kafka topic and consumes them back using a consumer group. You will need a running Kafka broker—`localhost:9092` is assumed throughout, but any Kafka-compatible broker works.

<Note>
  If you do not have a broker available locally, you can use `pkg/kfake` to run
  a fully in-process fake Kafka cluster. See the `examples/testing_with_kfake`
  directory for patterns.
</Note>

<Steps>

<Step title="Install the package">

Add franz-go to your Go module. The main client lives inside the `kgo` package within the `franz-go` module:

```bash
go get github.com/twmb/franz-go
```

Verify the entry appears in your `go.mod`:

```
require (
    github.com/twmb/franz-go v1.20.7
)
```

</Step>

<Step title="Create a client">

A single `kgo.Client` can produce and consume at the same time. You configure it with variadic option functions. The example below creates a client that joins the consumer group `"my-group-identifier"` and subscribes to the topic `"foo"`:

```go
package main

import (
    "context"
    "fmt"
    "sync"

    "github.com/twmb/franz-go/pkg/kgo"
)

func main() {
    seeds := []string{"localhost:9092"}

    // One client can both produce and consume!
    // Consuming can either be direct (no consumer group), or through a group.
    // Below, we use a group.
    cl, err := kgo.NewClient(
        kgo.SeedBrokers(seeds...),
        kgo.ConsumerGroup("my-group-identifier"),
        kgo.ConsumeTopics("foo"),
    )
    if err != nil {
        panic(err)
    }
    defer cl.Close()

    ctx := context.Background()
    _ = ctx // used in the steps below
}
```

<Note>
  Always `defer cl.Close()` so the client can flush pending records, commit
  offsets, and leave the consumer group cleanly before your program exits.
  Pass a `context.Context` to every blocking call so you can propagate
  cancellation and deadlines from your application logic.
</Note>

</Step>

<Step title="Produce records">

All record production goes through `Produce`. The callback lets you choose between asynchronous and synchronous semantics:

<CodeGroup>

```go async produce
// Produce is asynchronous by default. The callback fires when the broker
// acknowledges (or permanently fails) the record.
var wg sync.WaitGroup
wg.Add(1)
record := &kgo.Record{Topic: "foo", Value: []byte("bar")}
cl.Produce(ctx, record, func(_ *kgo.Record, err error) {
    defer wg.Done()
    if err != nil {
        fmt.Printf("record had a produce error: %v\n", err)
    }
})
wg.Wait()
```

```go sync produce
// ProduceSync blocks until all supplied records are acknowledged.
// It returns a slice of results; FirstErr() returns the first non-nil error.
record := &kgo.Record{Topic: "foo", Value: []byte("bar")}
if err := cl.ProduceSync(ctx, record).FirstErr(); err != nil {
    fmt.Printf("record had a produce error while synchronously producing: %v\n", err)
}
```

</CodeGroup>

`Produce` is the lower-level primitive and is preferred for high-throughput pipelines because it never blocks the caller. `ProduceSync` is convenient for one-off or low-frequency production where you want a simple error return, and it automatically un-lingers the partition to avoid any linger delay.

<Tip>
  For keys, set `record.Key = []byte("my-key")`. franz-go uses sticky
  partitioning by default (KIP-480/KIP-794), which keeps records in the same
  batch until the batch is full or flushed, maximizing compression and throughput.
</Tip>

</Step>

<Step title="Consume records">

Call `PollFetches` in a loop to receive batches of records. The call blocks until at least one record is available or the context is cancelled:

```go
for {
    fetches := cl.PollFetches(ctx)
    if errs := fetches.Errors(); len(errs) > 0 {
        // All errors are retried internally when fetching, but non-retriable
        // errors are returned from polls so that users can notice and take action.
        panic(fmt.Sprint(errs))
    }

    // We can iterate through a record iterator...
    iter := fetches.RecordIter()
    for !iter.Done() {
        record := iter.Next()
        fmt.Println(string(record.Value), "from an iterator!")
    }

    // or a callback function.
    fetches.EachPartition(func(p kgo.FetchTopicPartition) {
        for _, record := range p.Records {
            fmt.Println(string(record.Value), "from range inside a callback!")
        }

        // We can even use a second callback!
        p.EachRecord(func(record *kgo.Record) {
            fmt.Println(string(record.Value), "from a second callback!")
        })
    })
}
```

`fetches.Errors()` returns only non-retriable errors. Transient network errors and leader elections are retried transparently inside the client; you only see errors that require application-level intervention.

<Note>
  When you use `ConsumerGroup`, offsets are committed automatically after each
  successful `PollFetches` call by default. To take manual control of commits,
  disable auto-commit and use `cl.CommitRecords`, `cl.CommitUncommittedOffsets`,
  or `cl.CommitOffsets` directly. See the `examples/group_committing` example
  for all five commit strategies.
</Note>

</Step>

</Steps>

## Complete example

Putting all four steps together gives you a single file that produces one record and then consumes it back:

```go
package main

import (
    "context"
    "fmt"
    "sync"

    "github.com/twmb/franz-go/pkg/kgo"
)

func main() {
    seeds := []string{"localhost:9092"}

    // One client can both produce and consume!
    // Consuming can either be direct (no consumer group), or through a group.
    // Below, we use a group.
    cl, err := kgo.NewClient(
        kgo.SeedBrokers(seeds...),
        kgo.ConsumerGroup("my-group-identifier"),
        kgo.ConsumeTopics("foo"),
    )
    if err != nil {
        panic(err)
    }
    defer cl.Close()

    ctx := context.Background()

    // 1.) Producing a message
    // All record production goes through Produce, and the callback can be used
    // to allow for synchronous or asynchronous production.
    var wg sync.WaitGroup
    wg.Add(1)
    record := &kgo.Record{Topic: "foo", Value: []byte("bar")}
    cl.Produce(ctx, record, func(_ *kgo.Record, err error) {
        defer wg.Done()
        if err != nil {
            fmt.Printf("record had a produce error: %v\n", err)
        }
    })
    wg.Wait()

    // Alternatively, ProduceSync exists to synchronously produce a batch of records.
    if err := cl.ProduceSync(ctx, record).FirstErr(); err != nil {
        fmt.Printf("record had a produce error while synchronously producing: %v\n", err)
    }

    // 2.) Consuming messages from a topic
    for {
        fetches := cl.PollFetches(ctx)
        if errs := fetches.Errors(); len(errs) > 0 {
            // All errors are retried internally when fetching, but non-retriable errors are
            // returned from polls so that users can notice and take action.
            panic(fmt.Sprint(errs))
        }

        // We can iterate through a record iterator...
        iter := fetches.RecordIter()
        for !iter.Done() {
            record := iter.Next()
            fmt.Println(string(record.Value), "from an iterator!")
        }

        // or a callback function.
        fetches.EachPartition(func(p kgo.FetchTopicPartition) {
            for _, record := range p.Records {
                fmt.Println(string(record.Value), "from range inside a callback!")
            }

            // We can even use a second callback!
            p.EachRecord(func(record *kgo.Record) {
                fmt.Println(string(record.Value), "from a second callback!")
            })
        })
    }
}
```
